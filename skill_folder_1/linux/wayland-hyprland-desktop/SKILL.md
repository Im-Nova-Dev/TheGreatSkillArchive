# Wayland Hyprland Desktop

Practical workflow tuning and recovery for Hyprland on Arch-based systems, focused on real telemetry and failure modes rather than generic config snippets.

## When to use
- Blank session, no panels/wallpaper after Hyprland starts
- Monitor list wrong, duplicated, or missing outputs
- Workspace jump doesn't match expected monitor/rule assignment
- Floating windows or pin rules not applying, animations broken
- Per-app rules regressing and `hyprctl` reporting old rule keywords

## Procedure

### 0. New-session config audit (before any changes)
Run the Omarchy keybindings viewer to see what's actually live:
```bash
omarchy-menu-keybindings --print
```
This wraps `hyprctl -j binds` with keycode→symbol resolution and shows all active bindings with modifiers. Faster than grepping config files.

### 0b. Inspect Elephant/Walker launcher (Omarchy default)
Omarchy uses **Elephant** as a data provider for **Walker** (app launcher). Config paths:
- Walker: `~/.config/walker/config.toml`
- Elephant providers: `~/.config/elephant/*.toml`, `~/.config/elephant/menus/*.lua`
- System services: `elephant.service`, `app-walker@autostart.service` (user systemd)

To debug launcher behavior, inspect provider entries in Elephant menus — they can create apparent "key bindings" for search prefixes (e.g., typing "email" in Walker might launch Hey.com via a menu provider).

### 1. Normalize monitor/workspace binding
If workspaces appear on the wrong monitor or duplicated monitors show in `hyprctl monitors`, remove implicit `workspace = ...` rules and use explicit monitor-bound workspace binds.
1. List monitors and outputs
   text
   bash
   hyprctl monitors
   ```
2. Bind workspaces directly to each monitor using the monitor names from step 1:
   text
   hypr
   bind = SUPER, 1, workspace, 1
   bind = SUPER, 2, workspace, 2
   bind = SUPER, 3, workspace, 3
   workspace = 1, monitor:HDMI-A-1
   workspace = 2, monitor:HDMI-A-1
   ```
3. Remove redundant `workspace = 1`, `workspace = 2` fragments that lack a `monitor:` selector; they conflict with monitor rule inheritance.

### 2. Reconcile mixer selector state for audio
When pavucontrol shows multiple cards but no streams, or volume changes route unexpectedly, pavucontrol's Mixer drop-down is often still pinned to an old card.
- Set Mixer to the card that matches `hyprctl devices` audio sink, then reload sink profile.
Verify with `pactl get-card-profile` or by confirming active sink labels in pavucontrol.

### 3. Use rule types that match current Hyprland version
Hyprland renamed asymmetric rules to `windowrulev2` and `layerrulev2` starting around 0.53. If errors contain `windowrule = ...` or `layerrule = ...`, migrate those lines; the old keyword silently fails.
   text
   hypr
   windowrulev2 = float,class:kitty
   layerrulev2 = blur,class:waybar
   ```

### 4. Animation override guard
If animations default to the surprise generator after edits:
1. Restore an explicit `animations` block.
2. Check for duplicated keywords like `surprise` in autogenerated `*.override` files and remove them.
3. Recreate custom bezier curves after the session launches successfully.

### 5. Fix rule operator whitespace
Window/layer rule operators require immediate token attachment. `+. size` or `-. opacity` with a space breaks per-rule setters. Use `+:` / `-:` lines or `+.size` with no whitespace.
Inspect current keyword state with:
   text
   bash
   hyprctl keyword -p
   ```

### 6. Refresh without killing the layout
Run
   text
   bash
   hyprctl reload
```
If Waybar disappears after reload but Hyprland is still running, restart it:
   text
   bash
   pkill -f waybar && waybar &
```
If no outputs are reported after monitor changes, try
   text
   bash
   hyprctl dispatch reload
```

## Monitor scaling and keybinds
**Omarchy ships a dedicated helper:** `omarchy-hyprland-monitor-scaling-cycle`. It cycles forward by default and supports `--reverse` to go back down.

**Preferred fix for an accidental scale bump:** add symmetric bindings instead of manual edits.
```text
hypr
bindd = SUPER, CTRL, equal, Display scale up, exec, omarchy-hyprland-monitor-scaling-cycle
bindd = SUPER, CTRL, minus, Display scale down, exec, omarchy-hyprland-monitor-scaling-cycle --reverse
```

Current scale is visible in `/home/nova/.config/hypr/monitors.conf` as the last field on the active `monitor = ...` line, and live via `hyprctl monitors -j | jq -r '.[] | select(.focused == true).scale'`.

## Verification
- `hyprctl monitors` lists exactly the expected outputs with correct modes.
- `hyprctl workspaces` confirms workspace-to-monitor assignment.
- `hyprctl clients` shows per-app rules applied without `Invalid parameters`.
- Audio sink matrix in `pavucontrol` Mixer matches playback target.
- Animations transition without fallback to the surprise generator.

## Common Pitfalls
- Old `windowrule` / `layerrule` lines fail silently after Hyprland 0.53; they do not loudly error out.
- `+. size` whitespace breaks per-rule setters; unify operators as `+:` lines.
- `workspace = 1`, `workspace = 2` fragments conflict with monitor rule inheritance; prefer `workspace = monitor, <id>` or global binds.
- Mixer selector drift after reconnecting an audio card leaves all sinks seemingly silent.

## Input: keyboard, repeat, layout, and device routing
Use this when the keyboard layout is wrong, key repeat jittery, or an input device like a trackpoint/touchpad produces no events.

### 1. Inspect detected input devices
Run:
```bash
hyprctl devices
```
For per-device state and broader libinput info, inspect:
```bash
journalctl /usr/bin/hyprland --since "10 min ago" | rg -i "input|keyboard|pointer|touchpad|libinput"
```
Note the device name/numeric ID and the `libinput` category reported for each device.

### 2. Inspect currently active keyboard settings
```bash
hyprctl keyword -p | rg -i "kb_layout|kb_variant|kb_model|kb_options|kb_rules|kb_file|repeat_rate|repeat_delay"
```
If a previous session applied temporary overrides, `hyprctl reload` alone may not revert all of them. Clear affected keywords explicitly with empty values before restoring defaults:
```bash
hyprctl keyword kb_layout ""
hyprctl keyword kb_variant ""
```
Then reapply the explicit block from `hyprland.conf` and `hyprctl reload`.

### 3. Fix common keyboard layout issues
- **Layout not applied after hotplug:** some setups rely on XKB context files that are only loaded at compositor start. Ensure `kb_file` is set only if the file exists; remove it to fall back to XKB rules-based lookup:
  ```text
  hypr
  kb_layout = us,ru
  kb_variant = ,winkeys
  # Remove kb_file unless you explicitly need a custom xkb symbol file
  ```
- **CapsLock / numpad behavior drift:** set the behavior explicitly instead of relying on Xorg defaults:
  ```text
  hypr
  kb_options = caps:swapescape,numpad:mac
  ```
- **Absolute layout needed:** use `kb_layout` and `kb_variant` lists only; `kb_file` should point to a real file or be omitted. Avoid mixing `kb_file` with `kb_layout` unless necessary.

### 4. Stabilize key repeat
- Set both delay and rate in the same input block to avoid inherited or transient values:
  ```text
  hypr
  general {
    repeat_rate = 50
    repeat_delay = 250
  }
  ```
- If repeat still fires inconsistently on the login prompt vs session, also check the console parameters:
  ```bash
  localectl status
  ```
  The initial VT keyboard parameters are independent from Hyprland and can appear as “first key does not repeat.”

### 5. Per-device routing and sensitivity
To force one device to act as keyboard versus pointer, tune matching with `device[<name>]` blocks. Use exact or partial match quotes if the name contains spaces:
```text
hypr
device {
  name = "Lenovo TrackPoint Keyboard"
  kb_layout = us
}
device {
  name = "*-TouchPad
  natural_scroll = true
  tap_to_click = true
}
```
Matching is substring-based. If the device still does not match, remove quotes around substrings only if the config parser expects unquoted tokens:
```text
hypr
device {
  name = *-TouchPad
  natural_scroll = true
}
```
Use one-line minimal names first; quote when needed, then expand.

### 6. Instant temporary overrides
Test repeated key rate or caps behavior without editing files:
```bash
hyprctl keyword repeat_rate 70
hyprctl keyword repeat_delay 180
hyprctl keyword kb_options "caps:swapescape"
```
Note that capital `SUPER` and blanketed options may be overridden again on reload; save working values to `hyprland.conf` after confirming.

### 7. CapsLock behavior and Omarchy defaults (added from session debugging)
**Omarchy default:** `kb_options = compose:caps` (CapsLock = Compose key), NOT `caps:super`.

**Runtime verification:** `hyprctl devices -j` shows active keymap with `options: "compose:caps"` in keyboard objects.

**Common CapsLock modes:**
| Option | Behavior |
|--------|----------|
| `compose:caps` | CapsLock = Compose key (Omarchy default) |
| `caps:super` | CapsLock = extra Super key |
| `caps:swapescape` | CapsLock ↔ Escape |
| `caps:escape` | CapsLock = Escape |
| `caps:ctrl_modifier` | CapsLock = Ctrl when held |

**To make CapsLock an extra Super:** change `~/.config/hypr/input.conf`:
```ini
kb_options = caps:super
```
Then `hyprctl reload`.

**Pitfall:** `hyprctl keyword -p` may NOT show kb_options if not explicitly set in current session. Always verify with `hyprctl devices -j` for live state.

### 8. Debugging keyboard bindings end-to-end
When a key combo doesn't fire:
1. `hyprctl -j binds | jq -r '.[] | select(.key=="B") | "\(.modmask) \(.description) \(.dispatcher) \(.arg)"'` — find all bindings for a key
2. `hyprctl devices -j` — verify keyboard `options` field shows expected `kb_options` (e.g., `caps:super`)
3. Check both `~/.config/hypr/bindings.conf` AND `~/.local/share/omarchy/config/hypr/bindings.conf` + defaults in `~/.local/share/omarchy/default/hypr/bindings/`
4. If modifier is CapsLock: ensure `kb_options = caps:super` (not `compose:caps`). `compose:caps` consumes the key at XKB level; `modmask: 2` appears in binds but events never reach Hyprland.
5. After any `input.conf` change: `hyprctl reload` required (not just `hyprctl keyword`)

### 9. Modmask reference (from `hyprctl -j binds`)
| Mask | Modifiers |
|------|-----------|
| 0 | none |
| 1 | SHIFT |
| 2 | CAPSLOCK |
| 4 | CTRL |
| 8 | ALT |
| 64 | SUPER |
| 65 | SUPER+SHIFT |
| 68 | SUPER+CTRL |
| 72 | SUPER+ALT |
| 76 | SUPER+CTRL+ALT |
| 77 | SUPER+SHIFT+CTRL+ALT |

### 10. Recovery checklist
1. `hyprctl devices` → confirm expected device presence
2. `hyprctl keyword -p | rg -i "kb_|repeat_"` → identify stale overrides
3. Clear transient keywords explicitly, then reload and verify
4. Push bad device names through `device[]` blocks with `natural_scroll`, `tap_to_click`, or explicit `kb_layout`
5. For trackpoint/trackpad issues, inspect libinput quirks: `libinput list-devices`, look for `Kernel:` and `Change:` events
6. If the problem is hardware, swap devices or test in another TTY before adjusting config further

## Hypridle (Idle Daemon) Troubleshooting

Omarchy starts `hypridle` via `exec-once = uwsm-app -- hypridle` in `~/.local/share/omarchy/default/hypr/autostart.conf`. The user config lives at `~/.config/hypr/hypridle.conf` and controls:
- **Screen off / screensaver** (first `listener`)
- **Auto-lock** (second `listener`)
- **Suspend** (via `inhibit_sleep` and `before_sleep_cmd` / `after_sleep_cmd`)

**GUI "idle off" toggles often don't disable these listeners** — they only affect DPMS or power profiles. If the computer still sleeps/locks, check the config:

```bash
cat ~/.config/hypr/hypridle.conf
```

**Disable all idle actions (recommended fix):**
```bash
cat > ~/.config/hypr/hypridle.conf << 'EOF'
general {
    lock_cmd = omarchy-system-lock
    before_sleep_cmd = OMARCHY_LOCK_ONLY=true omarchy-system-lock
    after_sleep_cmd = sleep 1 && omarchy-system-wake
    inhibit_sleep = 3
}
# No listeners = no idle actions
EOF
omarchy-restart-hypridle
```

**Toggle temporarily (until next login):**
```bash
omarchy-toggle-idle
```

**Verify hypridle is running:**
```bash
pgrep -x hypridle
hyprctl devices -j  # check keyboard options not interfering
```
