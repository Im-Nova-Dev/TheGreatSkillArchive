# OSS & Community Monetization Workflow

## Purpose
Classify and format open-source/bounty intelligence so it lands as reusable teaching and skill updates, not scattered flat notes.

## When to Use
- Best intel for the run came from OSS bounty / sponsor sources.
- A run just created a file such as `2026-06-05-oss-bounty-opire-monetization.md`.
- You added a dated appendix to `open-source-bounty-sponsorship-routing` and need a sibling class skill.

## Class-Level Structure
Every substantive domain this intelligence system lands on should have:
1. A teaching umbrella under `career-and-monetization/`.
2. A concise SKILL.md with triggers and short steps.
3. At least one `references/` file with exact, dated evidence (numbers, platforms, ranges).

## Step 1: Classify the Intel
Match the dominant signal to the nearest existing umbrella:
- AI contracting / rates -> `ai-demand-driven-contract-packaging`.
- Market hiring trends / headcount -> `hiring-trend-radar`.
- OSS bounties / sponsorship / solo project revenue -> this workflow.

## Step 2: Write the Intel
Use the canonical intel shape, with emphasis on:
- platform names and bounty ranges ($, fee, competition).
- revenue model shifts (licensing -> trust/support).
- rate math: hourly effective vs bounty vs retainer.

## Step 3: Patch the Nearest Target Skill
Usually that is `open-source-bounty-sponsorship-routing`.
Add a dated appendix with new platform numbers, the revenue model framing, rate math, and implementation notes.

## Step 4: Wire a Reference File
Add `references/<topic>.md` with:
- platform names + exact ranges + date.
- source URLs or source names.
- how this evidence maps to existing skill steps.
