# Socratic Questioning Guide — Git and GitHub

Use questions instead of commands to build understanding.

## Core Principle

The goal is for learners to arrive at the answer themselves, not to memorize commands.

## Beginner Questions

1. What do you want to happen when you type `git add`?
2. Where does `git commit` save work: on disk, in memory, or on GitHub?
3. If you create a new branch, do you copy all your files?
4. What does `git status` check?
5. When should you push?

## Intermediate Questions

1. If two branches diverge, what do you expect `git merge` to do?
2. What is the difference between `git pull` and `git fetch`?
3. When would you choose `rebase` over `merge`?
4. What makes a good conflict resolution?
5. Why does a PR need a title and description?

## Advanced Questions

1. How would you protect a shared branch from accidental rewrite?
2. When is a merge commit the right choice versus squash?
3. How can you test whether a bug was introduced before or after a certain commit?
4. What does `git filter-repo` change, and why is it dangerous on shared history?
5. How can you automate part of your Git workflow?

## Conversational Pattern

1. Ask the question
2. Wait and accept uncertainty
3. If stuck, ask a smaller question
4. If still stuck, demonstrate and explain what changed
5. Loop back: “Now what would you try next?”

## Common Traps

- Avoid yes/no questions that end conversation
- Avoid asking “What is the command?”
- Ask “What do you want to happen?” instead

## Tips

- Relate Git actions to real-world actions: saving, mailing, branching, merging
- Use whiteboard diagrams for branch histories
- Ask learners to predict before they run commands
